# Frontend-архітектура сайту UAPP

Робочий документ: технологічний стек, структура коду, дизайн-система, CMS, документація та процес розробки (SDD) для реалізації редизайну uapp.group. Спирається на [бриф](task/uapp-redesign-brief.md) (потік 12 блоків, performance-бюджет, WCAG AA, hero-анімація) та [стайлгайд](brand-style-guide.md) (задані токени).

**Етапність.** Поточна ітерація — High-Fidelity прототип головної (desktop, EN). Архітектура нижче спроєктована так, щоб прототип без переписування виростав у продакшн-сайт (повна структура сторінок, CMS, адаптив) — це і є головний архітектурний критерій вибору рішень.

---

## 1. Принципи (випливають з брифу, не обговорюються окремо)

1. **Performance — це бренд-якість.** «Інженерна точність» і «regulated-grade» не читаються з сайту, який довго вантажиться. Ціль: LCP < 2.0s, CLS < 0.1, JS на сторінці — тільки там, де є інтерактив (hero-ефект, таби showcase, форма). Конкретні числа бюджету — відкрите питання брифу §11.3, до уточнення тримаємось цих.
2. **零 JS за замовчуванням.** Сайт контентний: 10 з 12 блоків головної — статичні. JavaScript отримують лише острови інтерактиву.
3. **Токени — єдине джерело стилю.** Жодного hex-коду чи font-family у компонентах: тільки семантичні токени зі стайлгайду. Це технічна гарантія «єдиного стилю» (розд. 4).
4. **Доступність за замовчуванням:** WCAG AA закладено вже в семантичні пари стайлгайду (розд. 4 стайлгайду); `prefers-reduced-motion` — обов'язковий fallback для hero та всіх scroll-анімацій (мандат брифу §7).
5. **Контент відділений від верстки** — навіть без CMS у першій ітерації (розд. 5).

## 2. Технологічний стек

### 2.1 Фреймворк: Astro

**Рекомендація — [Astro](https://astro.build) (v5+, static output).** Обґрунтування під наш кейс:

- **Islands architecture** — пряме втілення принципу «0 JS за замовчуванням»: сторінка рендериться в статичний HTML, а hero-ефект, таби showcase і форма підключаються як ізольовані острови зі своїм JS. Найкращий досяжний LCP для контентного сайту з одним важким ефектом.
- **Content Collections** — типізований контент (Markdown/JSON + Zod-схема) з коробки: у першій ітерації замінює CMS, у продакшні стає адаптером до headless CMS без зміни компонентів (розд. 5).
- **Фреймворк-агностичність островів** — hero-ефект пишемо на vanilla TS + WebGL без обгортки; якщо для окремого острова знадобиться React (наприклад, складна форма) — підключається точково.
- Прототип «розроблений за допомогою ШІ» (вимога брифу §10) з Astro працює добре: компонентна модель проста, `.astro`-файли близькі до HTML.

**Коли переглянути на користь Next.js:** якщо у продакшн-команди UAPP сильна React-експертиза і в roadmap з'являться app-подібні фічі (клієнтський портал, дашборди). Для маркетингового сайта-візитівки з блогом Next.js — надлишковий: тягне React-runtime на кожну сторінку і ускладнює performance-бюджет.

### 2.2 Решта стеку

| Шар | Рішення | Чому |
|---|---|---|
| Мова | TypeScript (strict) | Типізація контенту й пропсів — дешева страховка при AI-генерації коду |
| Стилі | Tailwind CSS v4 | Токени стайлгайду декларуються в `@theme` як CSS custom properties — один файл-джерело (розд. 4.1); utility-класи добре генеруються ШІ і легко ревʼюються |
| Hero-ефект | [OGL](https://github.com/oframe/ogl) або vanilla WebGL, окремий острів | three.js (~150 KB) — надлишковий для одного ефекту; OGL ~8 KB gzip. Ліниве підвантаження після FCP, статичний fallback-кадр для reduced-motion |
| Мікроанімації | CSS transitions + `@starting-style`; [Motion](https://motion.dev) точково для scroll-reveal | CSS покриває hover/focus безкоштовно; Motion (~5 KB) — тільки якщо CSS scroll-driven animations не вистачить |
| Шрифти | e-Ukraine Head / e-Ukraine self-hosted, woff2, subset latin, `font-display: swap` | Ваги та webfont-ліцензія — відкриті питання стайлгайду; до отримання ліцензії в прототипі використовувати локально, у продакшн не деплоїти без ліцензії |
| Форми | Статичний HTML + острів валідації; сабміт у serverless-функцію → CRM/email | Куди йдуть ліди — відкрите питання брифу §11.5; інтерфейс форми не залежить від відповіді |
| Хостинг | Cloudflare Pages / Vercel / Netlify — будь-який зі static-виводом | Рішення не блокує розробку; CI-прев'ю на кожен PR — обов'язково |
| Якість | ESLint + Prettier, Playwright (smoke + axe-core a11y), Lighthouse CI з бюджетом | Перевірка чек-листа брифу §11 автоматизується частково (розд. 7.3) |

## 3. Структура коду

```
site/                          # окрема тека в цьому ж репо (моно-репо з документацією)
├── src/
│   ├── styles/
│   │   └── tokens.css         # ЄДИНЕ місце значень: примітиви + семантика (розд. 4.1)
│   ├── components/
│   │   ├── primitives/        # Button, Link, Heading, Badge, Container, Section
│   │   └── blocks/            # 12 блоків головної: Hero.astro, PositioningBand.astro,
│   │                          #   TrustBar.astro, Expertise.astro, Solutions.astro, ...
│   ├── islands/               # інтерактив: hero-effect/, solutions-tabs.ts, lead-form.ts
│   ├── content/               # типізований контент (EN) + Zod-схеми (розд. 5.1)
│   │   ├── home/              # по файлу на блок: hero.json, expertise.json, cases.json ...
│   │   └── config.ts          # схеми колекцій
│   ├── layouts/Base.astro     # <head>, шрифти, метадані, skip-link
│   └── pages/index.astro      # збірка блоків у потік брифу §6
├── public/fonts/              # woff2 e-Ukraine (після вирішення ліцензії)
└── tests/                     # Playwright: smoke, axe, reduced-motion
```

Правила:

- **Блок = компонент.** Кожен із 12 блоків брифу §6 — один компонент у `blocks/`, який приймає контент пропсами з `content/` і не містить хардкод-текстів. Перестановка блоків (бриф дозволяє пропонувати зміни порядку) — це перестановка рядків у `index.astro`.
- **Primitives — єдині елементи UI.** Кнопка існує в одному місці; блоки не малюють власних кнопок/заголовків.
- **Острів ≠ блок.** `Hero.astro` — статична верстка (H1, sub, CTA, метрики) видима без JS; `islands/hero-effect` домальовує ефект поверх. Так LCP-контент не заручник WebGL.

## 4. Дизайн-система: як тримати єдиний стиль

Єдиний стиль тримається не дисципліною, а **конвеєром токенів**: значення існує в одному місці, все інше на нього посилається.

### 4.1 Конвеєр токенів

```
docs/brand-style-guide.md  →  src/styles/tokens.css (@theme)  →  компоненти (лише семантичні класи)
        (джерело правди)         (машинне віддзеркалення)          (жодних сирих значень)
```

- **Примітиви** (`--color-ultramarine-600: #011EFF`, шкала gray, base) — 1:1 зі стайлгайдом, розд. 1.
- **Семантика** (`--color-ink`, `--color-body`, `--color-muted`, `--color-accent`, `--color-surface`, `--color-border`) — 1:1 з таблицею пар «світла/темна поверхня» стайлгайду, розд. 4.
- **Темні секції** (напрямок B «Dark Precision» — темні блоки всередині світлої сторінки, не глобальна dark mode): секція отримує `data-surface="dark"`, всередині якої семантичні токени перевизначені на темну колонку таблиці. Компоненти всередині не знають, на якій поверхні стоять, — правило стайлгайду «ultramarine/600+ ніколи не текст на темному» стає непорушним технічно, бо компонент фізично не обирає колір сам.
- **Типографічна шкала і спейсинг** — теж токени: обмежений набір ступенів (напр., display/h1/h2/h3/body/caption + шкала відступів 4/8-базова). Референсна мова (Column, Cal.com: ієрархія масштабом типографіки, розділення повітрям) вимагає саме жорсткої шкали, а не довільних значень.

### 4.2 Синхронізація з Figma

- У Figma-концепті завести **Variables** з тими самими іменами, що в `tokens.css` (`ink`, `body`, `accent`, `surface/dark`...). Один словник імен: дизайнер і код говорять «accent», а не «отой синій».
- Стайлгайд лишається джерелом правди; зміна токена = PR у `brand-style-guide.md` → оновлення `tokens.css` і Figma Variables одним заходом.

### 4.3 Механізми контролю

- **Лінт-заборона сирих значень:** ESLint/Stylelint-правило проти hex-кодів і `px`-магії поза `tokens.css` — головний автоматичний бар'єр.
- **Сторінка-специмен** `/styleguide` (не в проді): усі primitives на світлій і темній поверхні поруч — швидка візуальна перевірка перед мержем; це ж прев'ю для клієнта.
- **axe-core у CI** ловить контрастні порушення, якщо хтось усе-таки обійшов токени.
- Storybook на поточному масштабі (одна сторінка, ~15 компонентів) — надлишковий; сторінки-специмена достатньо. Повернутись до питання, коли компонентів стане 40+.

## 5. Контент і CMS

### 5.1 Ітерація 1: без CMS — і це рішення, а не борг

Головна сторінка — **мандатний копірайт** (бриф §8): його не редагують маркетологи щотижня, його погоджують. Тому:

- Контент головної живе у `src/content/home/` як типізовані JSON/Markdown-файли, по файлу на блок; Zod-схема фіксує структуру (hero: h1 з дозволеного списку, sub, cta, метрики...).
- Зміна копірайту = PR з рев'ю — це **фіча**: історія погоджень у git, неможливо «випадково переписати позиціонування» з адмінки.

### 5.2 Продакшн: headless CMS для потокового контенту

Принцип розділення: **CMS — для контенту, що тече (Insights, кейси, вакансії, команда); репозиторій — для позиціонування** (головна, positioning band, expertise-копірайт).

**Рекомендація — [Sanity](https://www.sanity.io).** Під наш кейс:

- Structured content зі схемами в коді (TypeScript) — контент-модель версіонується в git поруч із сайтом; для кейсів під NDA легко зробити обов'язкові поля `anonymizedClient`, `ndaChecked`.
- Hosted: нуль DevOps для 60-особової сервісної компанії без власної контент-платформи; безкоштовного тарифу вистачає (3 користувачі, необмежені public-датасети).
- Live preview і Portable Text — редактори Insights бачать сторінку в реальних токенах до публікації.

**Альтернатива — [Payload](https://payloadcms.com):** якщо з'явиться вимога self-hosted / повного володіння даними (для fintech-клієнтів UAPP це може стати аргументом «їмо свій dogfood»). TypeScript-native, ліцензія MIT, але це +інфраструктура (БД, хостинг адмінки) на власному обслуговуванні.

**Чого не робити:** WordPress/монолітні CMS (тягнуть свій фронтенд і руйнують performance-бюджет), а також підключення CMS «на виріст» уже в прототипі — інтеграція з'являється разом з Insights/Cases-сторінками, коли буде продакшн-етап. Astro Content Layer дозволяє підмінити локальні файли на Sanity-loader без зміни компонентів — тому відкладання нічого не коштує.

## 6. Документація проєкту

Структура вже склалась (README — вхідна точка, `docs/` — фінальні документи, `research/` — робочі матеріали). Для фази розробки додаються три жанри:

1. **ADR — Architecture Decision Records** (`docs/adr/NNNN-назва.md`, формат MADR-light: Контекст → Рішення → Наслідки, 15–20 рядків). Фіксують рішення, які потім захочеться переграти: вибір Astro, вибір CMS, техніка hero-ефекту, доля webfont-ліцензії. Цей документ породить перші 2–3 ADR, коли рішення підтвердяться прототипом.
2. **Специфікації блоків** (`docs/specs/`) — робочий інструмент SDD, розд. 7.
3. **README сайту** (`site/README.md`) — тільки операційне: як запустити, як задеплоїти, як прогнати тести. Без переказу брифу — на нього лінк.

Правила ведення (успадковані + нові):

- Робочі документи — українською; контент сайту і **код** (імена, коментарі, commit-повідомлення за бажанням) — англійською.
- Не дублювати: кожен факт живе в одному документі, решта посилаються. CLAUDE.md не переказує документи — так само specs не переказують бриф, а цитують мандатний копірайт дослівно (він і є контент).
- Документ, що суперечить коду, гірший за відсутній: ADR і specs оновлюються тим самим PR, що і код, який їх зачіпає.

## 7. SDD — spec-driven development: так, і ось як саме

### 7.1 Чому підхід тут доречний

Проєкт уже фактично живе за SDD, ще без коду: бриф = продуктова специфікація з acceptance-чек-листом (§11), стайлгайд = специфікація стилю з обчисленими контрастами. Деліверабл — «прототип, розроблений за допомогою ШІ» — а саме для AI-генерації специфікація і є головним важелем якості: LLM реалізує рівно те, що зафіксовано, і чек-лист дає об'єктивний критерій приймання. Ризик проєкту — не «складний код», а **дрейф від мандату** (переписаний копірайт, крипто-естетика, зламаний контраст); специфікації + автоматичні перевірки цільово закривають саме його.

### 7.2 Формат: специфікація на блок

По одному файлу на блок головної: `docs/specs/01-header.md` … `12-final-cta.md`. Скелет:

```markdown
# Spec: 02 — Hero

## Мандат (з брифу §7–8 — не змінювати)
- H1: "Engineering for regulated finance." (обраний з трьох дозволених)
- Sub, CTA "Describe your challenge", trust-метрики — дослівно з брифу
- Сигнатурний ефект: метафора «рух грошей / обидва береги»; НЕ карусель

## Дизайн-рішення (пропозиція дизайнера — можна обговорювати)
- Композиція, техніка ефекту, поведінка на скролі...

## Токени
- Поверхня: light · H1: ink · CTA: accent-заливка + white ...

## Стани й поведінка
- reduced-motion: статичний кадр ефекту; hover CTA: accent-hover; фокус-стилі...

## Acceptance (виконується перед мержем)
- [ ] Копірайт побайтово збігається з мандатом
- [ ] Ефект вантажиться після FCP; LCP-елемент — H1, не canvas
- [ ] prefers-reduced-motion → без руху, контент повний
- [ ] axe: 0 порушень; контраст пар — за таблицею стайлгайду
```

Три секції мають різний статус — **мандат / пропозиція / механіка** — це прямо відтворює розподіл відповідальності з брифу §1 («ТЗ задає бачення, ефекти пропонують дизайнери») і знімає суперечки «а це можна міняти?».

### 7.3 Цикл роботи

1. **Spec** — пишеться/оновлюється спека блоку (людина; ШІ може драфтити з брифу).
2. **Погодження** — рев'ю секції «Дизайн-рішення» (мандатну частину не рев'юють — вона переписана з ТЗ).
3. **Генерація** — ШІ реалізує блок за спекою; спека — головний вхід промпту.
4. **Верифікація** — acceptance-чек-лист: автоматизована частина в CI (axe, Lighthouse-бюджет, Playwright-тест reduced-motion, тест на дослівність мандатного копірайту — рядки з `content/` порівнюються зі спекою), візуальна частина — очима на прев'ю.
5. **Фіксація** — якщо в ході реалізації рішення змінилось, тим самим PR оновлюється спека (спека, що бреше, гірша за відсутню).

Глобальний чек-лист брифу §11 («зчитується banking-first», «немає крипто-естетики»...) — це acceptance не блоку, а сторінки: проганяється на зібраній головній перед показом клієнту.

**Інструментарій:** на цьому масштабі достатньо markdown-спек + CI; формальні фреймворки (GitHub spec-kit тощо) додають церемонію, виправдану на багатокомандних продуктах — тут не потрібні. Якщо продакшн-етап розростеться — рішення переглянути окремим ADR.

## 8. Порядок дій

1. Каркас `site/` (Astro + Tailwind + tokens.css з стайлгайду) + сторінка-специмен `/styleguide`.
2. Спеки блоків 01–12 з брифу (мандатні секції — механічно, дизайн-секції — у міру опрацювання концепту у Figma).
3. Блоки за циклом SDD: спершу статичні (positioning band, expertise, cases), hero-ефект — окремим треком паралельно.
4. CI: прев'ю-деплой, axe, Lighthouse-бюджет, копірайт-тест.
5. Прототип головної = деліверабл ітерації. Далі (окремий етап): адаптив, решта сторінок, підключення Sanity для Insights/Cases/Careers.
